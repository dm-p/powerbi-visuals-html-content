# AGENTS.md

Minimal orientation for AI coding agents working in this repo.

## What this is

A Power BI custom visual (`htmlContent`, displayName "HTML Content (lite)") that renders column or measure values as sanitized HTML inside a Power BI report. The lite edition does not load remote content and allows a smaller subset of HTML tags. Build/package via `pbiviz` (powerbi-visuals-tools).

- Entry point: [src/visual.ts](src/visual.ts)
- Visual config: [pbiviz.json](pbiviz.json), [capabilities.json](capabilities.json)
- Style: [style/visual.less](style/visual.less)

## Setup

```
npm install
```

Node 20+ is assumed (see `@types/node` ^20). No other one-time setup is required.

## Commands

| Task | Command |
|---|---|
| Dev server (live-reload into Power BI Desktop) | `npm start` |
| Package the visual (`.pbiviz`) | `npm run package` |
| Lint | `npm run eslint` |
| Format check / write | `npm run prettier-check` / `npm run prettier-format` |
| Unit tests (vitest) | `npm test` |
| Watch / UI / coverage | `npm run test:watch` / `npm run test:ui` / `npm run test:coverage` |
| Integration tests (Playwright CSP harness) | `npm run test:integration` |
| Full suite (unit + integration + docs check) | `npm run test:all` |
| Cert-check bundle | `npm run cert-check` |
| Regenerate sanitization rules doc | `npm run docs:generate` (CI uses `docs:check`) |
| Regenerate UAT corpus | `npm run uat:generate` |

Default to `npm test` for fast feedback during edits. Run `npm run test:all` before declaring sanitization work complete — `docs:check` will fail CI if `docs/sanitization-rules.md` drifts from the source-of-truth lists.

## Repo layout

```
src/                   # TypeScript source
  visual.ts            # IVisual entry point
  view-model.ts        # data → render-model transform
  behavior.ts          # DOM behaviors (selection, scroll, links)
  sanitize-pipeline.ts # core HTML/CSS sanitization (DOMPurify wrapper + hooks)
  css-sanitizer.ts     # CSS parsing/sanitization (postcss)
  svg-payload-scan.ts  # SVG data: URI inspection (fail-closed scanner)
  domain-utils.ts      # styling resolution, jsdom bootstrap
  visual-constants.ts  # allowlists, denylists, DOM constants
  visual-settings.ts   # format pane bindings
style/visual.less      # visual stylesheet
test/                  # vitest unit tests (mirror src/ filenames)
  fixtures/            # corpus + lorem fixtures
test-integration/      # Playwright CSP regression harness
  csp-harness/corpus.ts # canonical malicious / clean payload corpus
test-uat/              # Power BI UAT workbook + CSV corpora
scripts/               # tsx scripts (sanitization-rules generator, UAT corpus generator)
docs/                  # human + agent documentation (see Knowledge stores)
```

## Knowledge stores

Read these before starting work in their domain — they carry institutional context that is not derivable from the code alone.

- [docs/sanitization-rules.md](docs/sanitization-rules.md) — generated reference for what HTML/CSS the visual currently allows or blocks, plus a Troubleshooting section. Source of truth lives in `src/visual-constants.ts` and the sanitizer; this doc is regenerated by `npm run docs:generate` and validated by `docs:check`.
- [docs/solutions/](docs/solutions/) — solved problems, organized by category (`security-issues/`, `performance-issues/`, etc.) with YAML frontmatter (`module`, `tags`, `problem_type`, `root_cause`). Bug fixes, security findings, best practices, and workflow patterns. Relevant when implementing or debugging in a documented area — searchable by frontmatter fields and tags.
- [docs/uat.md](docs/uat.md) — the manual UAT process; read before changing UAT corpus generation or workbook fixtures.
- [docs/brainstorms/](docs/brainstorms/) — parked design discussions awaiting resolution. Check before starting work that touches a brainstormed area.
- [docs/plans/](docs/plans/) — implementation plans for in-flight or completed multi-step work.

## Conventions

- TypeScript strict mode (see [tsconfig.json](tsconfig.json)). Prefer narrow types over `any`; the sanitizer in particular relies on type discrimination at API boundaries.
- Tests mirror source filenames (`src/foo.ts` → `test/foo.test.ts`). New source files should ship with at least one corresponding test file.
- Security-adjacent changes (anything in `sanitize-pipeline.ts`, `css-sanitizer.ts`, `svg-payload-scan.ts`, `visual-constants.ts`) must add both positive and negative test cases — the corpus in `test-integration/csp-harness/corpus.ts` is the model.
- When tightening a sanitizer rule, also grep `test-uat/corpus.csv`, `test-uat/lorem.csv`, and `test-uat/stylesheet.csv` for fixtures that may have drifted from clean to blocked (or vice versa).
- Potential commits should first be presented for the developer to review/approve.
- Commit messages follow conventional-commit prefixes (`fix:`, `feat:`, `docs:`, `tests:`, `chore:`, `refactor:`).
